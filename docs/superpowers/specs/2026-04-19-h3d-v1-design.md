## 1. Why a redesign

The previous `partverse_edit_v1` workflow (`partcraft.cleaning.v1` + `scripts/cleaning/{promote_to_v1,encode_del_latent,rebuild_v1_index}.py`) was multi-step and required orchestrating a pending list (`del_latent.txt`) between promote and encode. Concrete pain points:

1. Promote was split: `promote_to_v1` only **linked** assets and appended deletions to a pending list; a separate `encode_del_latent` step had to be scheduled later on a GPU machine.
2. Pending-list state was a third coordination point alongside pipeline outputs and dataset; failures were hard to recover.
3. Layout mixed all edit types in one tree (`<root>/<obj_id>/<edit_id>/`), making it cumbersome to subset by edit type for downstream training.

H3D_v1 collapses this into one command per edit type, leverages the work s6p/s6b already do, and organises the dataset by edit type at the top level.

## 2. Goals & non-goals

**Goals**
- One CLI per edit type, end-to-end, no pending list.
- Dataset is **self-contained** (uploadable as a tar without losing referential integrity).
- Hardlink-based deduplication within the dataset (one obj's SLAT + 5 fixed-view originals are stored once and shared across all edits of that obj).
- Sharded path layout (`<type>/<NN>/<obj_id>/<edit_id>/`) so per-machine partial outputs are disjoint and trivially mergeable.
- Reuse existing pipeline_v3 artefacts (s6p preview, s6b after.npz) — promote does **zero GPU rendering** in the common case.
- Filter by per-type required gates (`final_pass`-equivalent), with required-null treated as **fail** (not auto-pass).

**Non-goals (this revision)**
- No new VLM gating; reuse existing `edit_status.json`.
- No SS re-encoding for objects that already have a flux edit produced.
- No data loader class — the layout + jsonl manifests are the contract.

## 3. Dataset layout (`data/H3D_v1/`)

```
data/H3D_v1/
├── _assets/                                    # dataset-internal physical pool
│   └── <NN>/<obj_id>/
│       ├── object.npz                          # (slat_feats, slat_coords, ss) — physical copy
│       └── orig_views/
│           ├── view0.png                       # VIEW_INDICES[0] = 89 (back-left overhead)
│           ├── view1.png                       # VIEW_INDICES[1] = 90 (back-right overhead)
│           ├── view2.png                       # VIEW_INDICES[2] = 91 (front-left overhead)
│           ├── view3.png                       # VIEW_INDICES[3] = 100 (front-right overhead)
│           └── view4.png                       # VIEW_INDICES[4] = 8  (front upward)
│
├── deletion/<NN>/<obj_id>/<edit_id>/
│   ├── meta.json                               # physical (views.best_view_index = K)
│   ├── before.npz                              # → hardlink _assets/<NN>/<obj_id>/object.npz
│   ├── after.npz                               # physical (s6b encoding output)
│   ├── before.png                              # → hardlink _assets/<NN>/<obj_id>/orig_views/view{K}.png
│   └── after.png                               # → hardlink <pipeline_obj>/edits_3d/<edit_id>/preview_{K}.png

├── addition/<NN>/<obj_id>/<edit_id>/
│   ├── meta.json                               # views.best_view_index mirrors paired deletion's K
│   ├── before.npz                              # → hardlink deletion/<NN>/<obj_id>/<paired_del_id>/after.npz
│   ├── after.npz                               # → hardlink _assets/<NN>/<obj_id>/object.npz
│   ├── before.png                              # → hardlink deletion/<NN>/<obj_id>/<paired_del_id>/after.png
│   └── after.png                               # → hardlink _assets/<NN>/<obj_id>/orig_views/view{K}.png
│
├── modification/<NN>/<obj_id>/<edit_id>/
│   ├── meta.json
│   ├── before.npz                              # → hardlink _assets/<NN>/<obj_id>/object.npz
│   ├── after.npz                               # physical (TRELLIS edit output, per-edit unique)
│   ├── before.png                              # → hardlink _assets/<NN>/<obj_id>/orig_views/view{K}.png
│   └── after.png                               # → hardlink <pipeline_obj>/edits_3d/<edit_id>/preview_{K}.png
│
├── scale/      …  same shape as modification
├── material/   …
├── color/      …
├── global/     …
│
└── manifests/
    ├── deletion/<NN>.jsonl                     # written by pull_deletion (per shard)
    ├── addition/<NN>.jsonl
    ├── modification/<NN>.jsonl
    ├── scale/<NN>.jsonl
    ├── material/<NN>.jsonl
    ├── color/<NN>.jsonl
    ├── global/<NN>.jsonl
    └── all.jsonl                               # aggregated by build_h3d_v1_index
```

### Layout invariants

1. Every dataset path is keyed by `<NN>/<obj_id>` after the edit type — two shards never collide.
2. `_assets/<NN>/<obj_id>/` is the **single physical source** for that obj's "before" SLAT and 5-view originals. All edits of that obj hardlink in.
3. Deletion's `after.npz` (s6b output) is the physical source for the paired addition's `before.npz` (hardlinked).
4. No `*.glb` in the dataset (decided 2026-04-19: dataset is npz + png only; glb stays in pipeline outputs for visualization).
5. Hardlinks only work within a single FS. Pipeline outputs and dataset root must live on the same FS; on cross-FS the promoter falls back to `shutil.copy2` (logged warning, not an error). With pipeline outputs and `data/H3D_v1/` both on CFS, this is a non-issue.

## 4. Why this is "free" (zero promote-time GPU in the common case)

Pipeline_v3 already produces every byte we need:

| dataset slot | source in pipeline | who wrote it |
|---|---|---|
| `_assets/.../object.npz` (slat_feats + slat_coords + ss) | any flux edit's `before.npz` (symlink-shared per obj) | s4 trellis (`refiner.export_pair_shared_before`) |
| `_assets/.../orig_views/view{i}.png` | any addition edit's `preview_{i}.png` | s6p (`render_trellis_previews_for_object`, addition branch) |
| `deletion/.../after.npz` | `<edit>/after.npz` (encoded from `after_new.glb`) | s6b — but **pipeline does not run s6b automatically**; `pull_deletion` invokes `link_slat_assets_for_object` itself for any accepted deletion missing `after.npz` |
| `deletion/.../after.png` (single K view) | `<edit>/preview_{K}.png`, K = `views.best_view_index` | s6p (deletion branch, renders from `after_new.glb`) |
| `flux/.../after.npz` | `<edit>/after.npz` from s4 | trellis refiner |
| `flux/.../after.png` (single K view) | `<edit>/preview_{K}.png`, K = `views.best_view_index` | s6p (TRELLIS branch) |

### "Before" 5-view originals — canonical processing

User constraint (2026-04-19): the per-object before image must be picked from the per-view raw PNG bytes packed in `<images_root>/<NN>/<obj_id>.npz`, then processed exactly the same way `phase1/overview.png` does (alpha-composite onto white background) so that before/after at the same view index overlay pixel-for-pixel.

The canonical implementation is `partcraft.render.overview.load_views_from_npz(image_npz, VIEW_INDICES)`:

```python
def load_views_from_npz(images_npz, view_indices):
    z = np.load(images_npz, allow_pickle=True)
    # cv2.imdecode each {idx:03d}.png by index
    # if RGBA: composite onto white
    return list_of_BGR_arrays, list_of_frame_dicts
```

s6p's TRELLIS-preview path already calls this function once per object; addition's `preview_{0..4}.png` is exactly its output (since addition's `after = original obj`).

`_ensure_object_views(obj_id)` therefore prefers reusing s6p's output and only falls back to the in-process call when no addition edit exists:

```python
def _ensure_object_views(obj_dir, image_npz, dst_dir):
    if all((dst_dir / f"view{i}.png").is_file() for i in range(5)):
        return
    # Path A: reuse any addition edit's preview (s6p already produced these)
    for add_dir in sorted((obj_dir / "edits_3d").glob("add_*")):
        if all((add_dir / f"preview_{i}.png").is_file() for i in range(5)):
            for i in range(5):
                shutil.copy2(add_dir / f"preview_{i}.png", dst_dir / f"view{i}.png")
            return
    # Path B: invoke load_views_from_npz directly (CPU-only, milliseconds)
    from partcraft.render.overview import VIEW_INDICES, load_views_from_npz
    imgs, _ = load_views_from_npz(image_npz, VIEW_INDICES)
    for i, img in enumerate(imgs):
        cv2.imwrite(str(dst_dir / f"view{i}.png"), img)
```

Both paths produce byte-identical results because they execute the same processing pipeline.

**Note (2026-04-19, after shard08 inspection)**: pipeline_v3 currently does *not* run s6b in the standard flow (only s5b → gate_a → s6p → gate_e). Deletion's `after.npz` is therefore typically missing on disk; `pull_deletion` reuses `partcraft.pipeline_v3.mesh_deletion.link_slat_assets_for_object` to encode it on demand. This is GPU-bound on first run for each accepted deletion edit (~5–30 s per edit on Blender + DINOv2 + SLAT encoder), and a no-op on subsequent runs. `pull_flux` and `pull_addition` remain pure IO.

### Edge cases requiring computation

- **obj has only deletion edits, no addition edit**: orig_views path A misses → path B (CPU) takes over.
- **obj has only deletion edits, no flux edit ever produced**: `object.npz` ss missing. Fallback: load `slat_dir/<NN>/<obj_id>_*.pt` and run `encode_ss` (single GPU call). In practice rare because phase1 always runs and emits flux candidate edits.

## 5. Filter rules

Read `<pipeline_obj_dir>/edit_status.json` (or per-edit qc fields) — do not re-run gates.

| CLI | accept iff | required-null behaviour |
|---|---|---|
| `pull_deletion`  | `gate_A.pass == True` | `gate_A == None` ⇒ **reject** |
| `pull_flux`      | `gate_A.pass == True AND gate_E.pass == True` | any required gate `None`/missing ⇒ **reject** |
| `pull_addition`  | paired deletion exists at `data/H3D_v1/deletion/<NN>/<obj_id>/<paired_del_id>/after.npz` | otherwise skip (will be picked up next run) |

Gate C is optional everywhere (treated as "if present must pass; if absent ignore"). This matches the user's earlier decision that gate C is not always run.

## 6. CLIs

All CLIs accept `--pipeline-cfg <yaml> --shard NN --dataset-root data/H3D_v1` and resolve pipeline output paths via the existing `partcraft.utils.config` loader so they pick up `images_root`, `slat_dir`, etc. from the same yaml the pipeline ran with.

### 6.1 `scripts/cleaning/h3d_v1/pull_deletion.py`

```bash
python -m scripts.cleaning.h3d_v1.pull_deletion \
    --pipeline-cfg configs/pipeline_v3_shard08_test20b_gateQ.yaml \
    --shard 08 \
    --dataset-root data/H3D_v1 \
    --device cuda:0 \
    --blender /usr/local/bin/blender \
    --workers 8
```

Per shard, end-to-end:
1. Iterate all `del_*` edits under the pipeline output for that shard.
2. Filter by `gate_A.pass == True`.
3. For each accepted edit:
   - **If `<edit_dir>/after.npz` is missing**, run `link_slat_assets_for_object` on `after_new.glb` to encode it (Blender → DINOv2 → SLAT encoder → SS encoder). GPU-bound, idempotent.
   - `_ensure_object_npz(obj_id)` — write `_assets/<NN>/<obj_id>/object.npz` if missing.
   - `_ensure_object_views(obj_id)` — write `_assets/<NN>/<obj_id>/orig_views/view{0..4}.png` if missing.
   - `mkdir -p deletion/<NN>/<obj_id>/<edit_id>`.
   - Hardlink `before.npz` from `_assets/.../object.npz`.
   - Hardlink `after.npz` from `<edit_dir>/after.npz`.
   - Resolve `K = views.best_view_index` (pipeline_v3 mask argmax; see §8).
   - Hardlink `before.png` from `_assets/.../orig_views/view{K}.png`.
   - Hardlink `after.png` from `<edit_dir>/preview_{K}.png`.
   - Write `meta.json`.
4. Append entry to `manifests/deletion/<NN>.jsonl`.

GPU is touched on every accepted deletion that lacks `after.npz` (typical first-run case) and by the rare `_ensure_object_npz` SS-fallback. Subsequent reruns of the same shard are pure IO.

### 6.2 `scripts/cleaning/h3d_v1/pull_flux.py`

```bash
python -m scripts.cleaning.h3d_v1.pull_flux \
    --pipeline-cfg configs/pipeline_v3_shard08_test20b_gateQ.yaml \
    --shard 08 \
    --types modification scale material color global \
    --dataset-root data/H3D_v1 \
    --workers 8
```

Pure CPU/IO. For each accepted flux edit:
- `_ensure_object_npz(obj_id)` — for flux this is trivially "copy `<edit_dir>/before.npz`" (it already has the full triplet).
- `_ensure_object_views(obj_id)` — see §4.
- Hardlink `before.npz` ← `_assets/.../object.npz`.
- Hardlink `after.npz` ← `<edit_dir>/after.npz`.
- Hardlink `before.png` ← `_assets/.../orig_views/view{K}.png`, `K = views.best_view_index` (`DEFAULT_FRONT_VIEW_INDEX` for `global`).
- Hardlink `after.png` ← `<edit_dir>/preview_{K}.png`.
- Write `meta.json`, append `manifests/<type>/<NN>.jsonl`.

`--types` defaults to all five flux types. Run subsets to parallelise across machines.

### 6.3 `scripts/cleaning/h3d_v1/pull_addition.py`

```bash
python -m scripts.cleaning.h3d_v1.pull_addition \
    --pipeline-cfg configs/pipeline_v3_shard08_test20b_gateQ.yaml \
    --shard 08 \
    --dataset-root data/H3D_v1
```

Pure hardlink. For each addition edit whose paired deletion is already in the dataset:
- Hardlink `before.npz` ← `deletion/<NN>/<obj_id>/<paired_del_id>/after.npz`.
- Hardlink `after.npz` ← `_assets/.../object.npz`.
- Hardlink `before.png` ← `deletion/<NN>/<obj_id>/<paired_del_id>/after.png` (K mirrored from paired deletion).
- Hardlink `after.png` ← `_assets/.../orig_views/view{K}.png`.
- Write `meta.json`, append `manifests/addition/<NN>.jsonl`.

If the paired deletion isn't promoted yet, skip silently (next run will pick it up).

### 6.4 `scripts/cleaning/h3d_v1/build_h3d_v1_index.py`

```bash
python -m scripts.cleaning.h3d_v1.build_h3d_v1_index \
    --dataset-root data/H3D_v1 \
    [--validate]
```

Aggregates `manifests/<type>/*.jsonl` into `manifests/all.jsonl`. With `--validate`, walks every entry and verifies hardlink targets exist + npz files load.

### 6.5 `scripts/cleaning/h3d_v1/pack_shard.py`

```bash
python -m scripts.cleaning.h3d_v1.pack_shard \
    --dataset-root data/H3D_v1 \
    --shard 08 \
    --out releases/H3D_v1__shard08.tar
```

Creates a tarball containing exactly that shard's slice (`_assets/<NN>/`, every `<type>/<NN>/`, `manifests/<type>/<NN>.jsonl`). `tar` preserves hardlinks within the archive, so the tarball is self-contained and deduplicates internally.

## 7. Concurrency & locking

Multiple `pull_*` runs may touch the same `_assets/<NN>/<obj_id>/` (e.g. `pull_deletion` and `pull_flux` running in parallel on the same shard). Lock by:

```python
import fcntl
lock_path = dataset_root / "_assets" / shard / obj_id / ".lock"
lock_path.parent.mkdir(parents=True, exist_ok=True)
with open(lock_path, "w") as f:
    fcntl.flock(f, fcntl.LOCK_EX)
    # ensure object.npz + orig_views/
```

Each per-edit promote is independent (writes its own dir); no extra locking needed.

## 8. `meta.json` schema (v3, final)

Minimal, release-oriented layout.  Per-edit bookkeeping (part ids,
labels, counts) and promoter-side metadata (git sha, config path,
timestamp, pair links) are intentionally omitted from the released
file — the former is pipeline-internal, the latter lives in
``manifests/_internal/promote_log.jsonl`` (see below) for local audit
only and is excluded from ``pack_shard`` tarballs.

```json
{
  "edit_id": "mat_<obj_id>_003",
  "edit_type": "material",
  "obj_id": "bdd36c94f3f74f22b02b8a069c8d97b7",
  "shard": "08",
  "schema_version": 3,

  "instruction": {
    "prompt":           "Change the material of the entire bowl to polished obsidian glass.",
    "object_desc":      "A wooden bowl composed of multiple structural components...",
    "target_part_desc": "wooden bowl component highlighted in red ...",
    "after_desc":       "A bowl made entirely of polished obsidian glass ...",
    "edit_params":      {"target_material": "polished obsidian glass"}
  },

  "views": {
    "best_view_index": 2
  },

  "quality": {
    "final_pass":      true,
    "alignment_score": 1.0,
    "quality_score":   0.8
  },

  "lineage": {
    "source_dataset":   "partverse",
    "pipeline_version": "v3"
  }
}
```

### Block reference

* **top-level** — stable identifiers and shard key only.
  ``schema_version == 3`` at all times for this layout.

* **instruction** — the edit effect as the model should see it.
  ``prompt`` is always present; ``object_desc`` + ``target_part_desc``
  carry context.  Type-scoped extras: ``new_parts_desc`` for
  ``modification``, ``after_desc`` / ``edit_params`` for the flux
  group when non-empty.  Pipeline-internal part bookkeeping
  (``part_labels``, ``n_parts_selected``) is **not** emitted.
  del↔add pairing is derivable from the edit_id convention
  (``del_<obj>_NNN`` ↔ ``add_<obj>_NNN``).

* **views.best_view_index** — integer in ``0..4`` identifying the
  single most-informative view for this edit.  Each edit ships
  exactly one ``before.png`` + one ``after.png`` hardlinked at
  this K (K maps to ``VIEW_INDICES`` in ``partcraft/render/overview.py``).
  Selection order:

  1. Non-addition edits with a selected-part mask → pipeline_v3
     pixel-mask argmax from ``edit_status.json.gates.A.vlm.best_view``.
  2. Addition edits → mirror the paired deletion's best_view.
  3. Global edits → fixed ``DEFAULT_FRONT_VIEW_INDEX``
     (defined in ``partcraft.cleaning.h3d_v1.promoter``;
     currently ``4`` = front upward, yaw +22° pitch +52°).
  4. Fallback on missing status → ``DEFAULT_FRONT_VIEW_INDEX``.

* **quality** — pass flag + two semantic scores.  ``alignment_score``
  is gate A (text/image alignment); ``quality_score`` is gate E (3D
  render quality).  ``final_pass`` is the AND of all active gates.

* **lineage** — upstream source only.  Just two fields:
  ``source_dataset`` (e.g. ``partverse``) and ``pipeline_version``
  (e.g. ``v3``).

### Internal promote log (not shipped)

Every successful ``promote_*`` appends one JSONL record to
``manifests/_internal/promote_log.jsonl``:

```json
{"edit_id": "del_...", "edit_type": "deletion", "obj_id": "...",
 "shard": "08", "promoted_at": "2026-04-20T06:50:22Z",
 "pipeline_version": "v3",
 "pipeline_config":  "configs/pipeline_v3_shard08.yaml",
 "pipeline_git_sha": "abc1234",
 "source_dataset":   "partverse"}
```

This file stays local.  ``pack_shard`` packs ``_assets/<NN>/``,
``<edit_type>/<NN>/``, ``manifests/<edit_type>/<NN>.jsonl`` and
``manifests/all.jsonl``; it deliberately does **not** include
``manifests/_internal/``.

## 9. Multi-machine workflow

Pipeline outputs live on shared CFS, so any machine that mounts CFS can run any `pull_*`. Two viable workflows:

**Workflow A — central machine drives everything**
- One CPU machine mounts CFS + dataset_root.
- Runs `pull_deletion --shard NN`, `pull_flux --shard NN`, `pull_addition --shard NN` for every shard sequentially or in parallel.
- Then `build_h3d_v1_index --validate`.
- (No GPU usage in the common case — see §4.)

**Workflow B — per-shard parallel across machines**
- Each machine runs `pull_deletion / pull_flux / pull_addition` for its assigned shards.
- Because every dataset path is shard-keyed, machines never write to overlapping directories.
- Central machine runs `build_h3d_v1_index --validate` after all shards finish.

For cloud upload: `pack_shard --shard NN` per shard → tarballs preserve hardlinks → upload tarballs → cloud-side extract reconstructs the full dataset. Avoid uploading the directory tree directly; most object stores don't preserve hardlinks.

## 12. Open questions / risks

- **glb omission cost**: deletion has `after_new.glb` for downstream consumers; not having it in the dataset means consumers wanting mesh have to keep CFS access. Acceptable per user decision 2026-04-19.
- **Cross-FS hardlink**: if dataset_root is moved off CFS later, hardlinks break and we'd need to flatten via `cp -al` then re-tar. Not an issue while everything stays on CFS.
- **Filter strictness for null gates**: previous v1 treated None as pass; H3D_v1 treats required-null as fail. This will reduce dataset size by N% (TBD via dry-run report).

## 13. Acceptance test (smoke)

1. Run `pull_deletion --shard 08` on the existing shard08 outputs.
2. Run `pull_flux --shard 08`.
3. Run `pull_addition --shard 08`.
4. Run `build_h3d_v1_index --validate` — must report 0 broken hardlinks, 0 unloadable npz.
5. Spot-check 3 random `<type>/<NN>/<obj_id>/<edit_id>/`:
   - `before.png` and `after.png` overlay correctly (same camera `K = views.best_view_index`, same background).
   - `before.npz` and `after.npz` load via `np.load` and contain expected keys.
   - For an addition: confirm its `before.npz` inode equals the paired deletion's `after.npz` inode (`stat -c %i`).
6. Run `pack_shard --shard 08`, verify tarball size ≈ deduplicated content size, and `tar -tvf | head` shows hardlink markers (`h` flag).
